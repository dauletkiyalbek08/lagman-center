# Lagman Center — запуск и подключение Supabase

Сайт halal-кафе «Lagman Center» (Щучинск): лендинг, меню, два способа заказать
(по QR со стола и доставкой), бронирование столов и панели персонала
(админ / кухня / курьер) с обновлением заказов в реальном времени.

## Два способа заказать

| | **В зале (QR на столе)** | **Доставка** |
|---|---|---|
| Регистрация | не нужна | обязательна (телефон + пароль) |
| Как попасть | сканирует QR → `/t/<код>` → меню | обычное меню на сайте |
| Что видит персонал | заказ с номером стола | адрес и телефон клиента |
| Оплата | на кассе при выходе | при получении (курьеру) |
| Курьер | не участвует | забирает готовый заказ |

QR-коды столов печатаются в панели: **/admin → Столы → «Печать QR-кодов»**.
Гость наводит камеру, видит меню, собирает корзину и жмёт «Отправить на кухню» —
заказ появляется у админа и на кухне с пометкой «Стол №N», а стол на карте зала
сразу становится занятым.

## Быстрый старт (демо-режим, без бэкенда)

```bash
npm install
npm run dev
```

Откройте http://localhost:3000. Пока Supabase не подключён, сайт работает в
**демо-режиме**: меню берётся из встроенного сида, заказы и брони сохраняются
в localStorage браузера, а «реальное время» между вкладками обеспечивает
BroadcastChannel. Откройте, например, `/admin` и сайт в соседних вкладках и
оформите заказ — он появится на доске без перезагрузки. Панели персонала в
демо-режиме открыты без входа.

## Подключение Supabase (боевой режим)

> **Проект уже создан и настроен:** `lagman-center` (регион Франкфурт).
> Схема применена, меню залито, гостевой заказ и бронь проверены на живой базе.
> Ключи для Vercel и локального `.env.local`:
>
> ```
> NEXT_PUBLIC_SUPABASE_URL=https://jzpheztyxkonzpwcqylc.supabase.co
> NEXT_PUBLIC_SUPABASE_ANON_KEY=<anon key из Project Settings → API Keys>
> ```
>
> Инструкция ниже — на случай, если проект придётся пересоздать с нуля.

1. Создайте проект на [supabase.com](https://supabase.com) (Free-плана достаточно).
2. Откройте **SQL Editor → New query**, вставьте содержимое файла
   [`supabase/schema.sql`](supabase/schema.sql) целиком и нажмите **Run**.
   Скрипт создаст таблицы, RLS-политики, realtime-публикацию, функции
   оформления заказа/брони и сид-данные (заведения + меню).
3. В **Authentication → Sign In / Up → Email** отключите «Confirm email»
   (иначе каждому клиенту придётся подтверждать почту письмом).
4. Скопируйте `.env.example` в `.env.local` и заполните из
   **Project Settings → API**:

   ```
   NEXT_PUBLIC_SUPABASE_URL=https://<project>.supabase.co
   NEXT_PUBLIC_SUPABASE_ANON_KEY=<anon key>
   ```

5. Перезапустите `npm run dev`.

### Вход клиентов и персонала

Форма входа одна (`/login`): **клиенты входят по номеру телефона**, сотрудники —
по email. Различает их сама форма: если в поле есть «@» — это email сотрудника,
иначе номер.

Регистрация клиента (`/register`) — только номер, имя, пароль и (по желанию)
адрес. Ни SMS, ни писем: номер играет роль логина, а учётку заводит функция БД
`register_customer()`. Так сделано намеренно — обычный `auth.signUp()` считает
логин почтой и пытается отправить письмо на несуществующий адрес
`7707…@phone.lagmancenter.kz`: письмо не доходит, сессия не выдаётся, а почтовый
лимит проекта выгорает впустую.

Логины сотрудникам выдаёт администратор — в панели
**/admin → Персонал → «Добавить сотрудника»** (email, пароль, роль).
Учётка работает сразу, письма подтверждения не нужны.

Готовые учётки (созданы схемой — **смените пароли перед боевым запуском**,
кнопка «Пароль» в той же вкладке):

| Роль | Логин | Пароль |
|---|---|---|
| Администратор | `admin@lagmancenter.kz` | `Lagman-Admin-2026` |
| Кухня | `kitchen@lagmancenter.kz` | `Lagman-Kitchen-2026` |
| Курьер | `courier@lagmancenter.kz` | `Lagman-Courier-2026` |

Вход — на `/login` (в шапке сайта иконка человечка, в подвале «Вход для персонала»).
После входа сотрудник попадает сразу в свою панель.

| Роль | Доступ |
|---|---|
| `admin` | `/admin`: заказы, столы и QR-коды, меню, брони, сотрудники, настройки доставки |
| `kitchen` | `/kitchen`: заказы «Готовится» → кнопка «Готово» (у заказов в зале виден номер стола) |
| `courier` | `/courier`: только доставка: «Готов к выдаче» → «Взял в доставку» → «Доставлен» |
| `customer` | клиент доставки: свои заказы и адрес по умолчанию |

### Столы и карта зала

**/admin → Столы** — карточка на каждый стол: свободен (зелёный), заняты гости
(красный), забронирован (жёлтый). Стол становится занятым сам, когда с его QR
приходит заказ; освободить можно кнопкой. Там же — «Добавить стол», QR-код
одного стола (скачать PNG) и «Печать QR-кодов» — лист карточек на все столы.

Бронь сажают за конкретный стол в **/admin → Брони** (выпадающий список) —
после этого стол подсвечивается на карте как забронированный.

### Настройки доставки

**/admin → Настройки**: стоимость доставки, бесплатная доставка от суммы,
минимальная сумма заказа. Клиент видит доставку отдельной строкой в корзине, но
считается она **на сервере** (в `create_order`) — подделать из браузера нельзя.

Предоплаты пока нет: доставку оплачивают при получении. Чтобы сайт сам подставлял
сумму в Kaspi и сам узнавал об оплате, заведению нужен **Kaspi Business (Kaspi
Pay)** — договор на ИП/ТОО, API-ключ и webhook. Схема к этому готова: у заказа
уже есть `payment_status` и `delivery_fee`.

### Меню и фотографии

В панели **/admin → Меню**: «Добавить блюдо» — название, описание, цена, категория
(выбрать из списка или ввести свою: «Напитки», «Шашлыки», «Европейская кухня» — что угодно),
фото (перетащить файл или выбрать — оно улетит в Supabase Storage, бакет `menu`)
и переключатель «В продаже». Меню на сайте обновляется сразу, без перезагрузки.

Жизненный цикл заказа: `new → cooking → ready → delivering → delivered`
(или `cancelled` админом). Все панели подписаны на Supabase Realtime — заказ
перескакивает между экранами без перезагрузки.

### Как устроен доступ к данным (важно при доработках)

- **Заказ и бронь создаются функциями** `create_order()` и
  `create_reservation()` (`security definer`), а не прямым `insert` из
  браузера. Причина: гость не имеет права читать чужие заказы, а
  `insert ... returning` требует ещё и SELECT-права — прямой insert падал бы
  на RLS. Заодно функция считает сумму по ценам **из базы** (цену и стоимость
  доставки нельзя подделать из devtools) и пишет заказ вместе с позициями
  атомарно. Она же не даст заказать доставку без входа (`auth.uid() is null`).
- **Заказ в зале без регистрации** работает: гостю отдаётся только статус
  заказа — через функцию `order_status(uuid)` (id заказа неугадываем).
- **Столы гость не читает.** В строке стола лежит код из QR, поэтому таблица
  `tables` открыта только персоналу, а гость получает свой стол по коду через
  `table_by_code()`.
- **`create_auth_user()` недоступна снаружи.** Supabase по умолчанию раздаёт
  EXECUTE на новые функции ролям `anon`/`authenticated` — права отозваны явно,
  иначе кто угодно мог бы завести учётку с произвольным email.
- **Персонал читает все заказы**, хотя менять статус может лишь в своей зоне
  (кухня: `cooking → ready`, курьер: `ready → delivering → delivered`).
  Так сделано намеренно: Realtime присылает событие только тому, кто вправе
  прочитать **новую** версию строки, — иначе отменённый заказ навсегда завис
  бы на доске кухни.
- **Курьер забирает заказ гонко-безопасно**: обновление идёт с условием
  `status = 'ready'`, поэтому второй курьер получит «Заказ уже забрал другой
  курьер», а не перехватит доставку.

## Деплой на Vercel

1. Запушьте репозиторий на GitHub.
2. [vercel.com](https://vercel.com) → **Add New Project** → импортируйте репозиторий.
3. В **Environment Variables** добавьте `NEXT_PUBLIC_SUPABASE_URL` и
   `NEXT_PUBLIC_SUPABASE_ANON_KEY`.
4. **Deploy**.

## Замена фотографий

Фото блюд лежат в `public/images/` (стоковые плейсхолдеры с Unsplash).
Замените файлы на фотографии заказчика с теми же именами — или меняйте
`image_url` блюда через панель администратора (подойдёт любая https-ссылка).

## Структура

- `src/app/(site)` — публичные страницы (главная, меню, доставка, бронь…)
- `src/app/(staff)` — панели персонала `/admin`, `/kitchen`, `/courier`
- `src/lib/data.ts` — слой данных: Supabase или локальное демо-хранилище
- `src/lib/demo-store.ts` — демо-режим (localStorage + BroadcastChannel)
- `supabase/schema.sql` — вся схема БД с RLS и сидами
