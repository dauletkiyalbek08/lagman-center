# 🍢 Промпт для студии: сайт «Lagman Center»

Собери сайт для **halal-кафе «Lagman Center»** (г. Щучинск, Казахстан). Это лендинг + онлайн-меню + оформление доставки + бронирование столов + **рабочая панель для персонала** (админ / кухня / курьер) с обновлением заказов в реальном времени.

Дизайн — тёмная тема с красным акцентом, как в референсе ниже. Весь текст интерфейса — **на русском языке**.

---

## 1. Рекомендуемый стек

- **Next.js (App Router) + TypeScript + Tailwind CSS** — сайт и панели в одном приложении
- **Supabase** — Auth (аккаунты + роли), Postgres (заказы/меню/брони), **Realtime** (живая доска заказов)
- **Vercel** — деплой
- **shadcn/ui** или простые Tailwind-компоненты — для форм и карточек
- **Lucide-иконки**
- (Опционально, фаза 2) **Telegram Bot API** для уведомлений персоналу

Одно приложение, роутинг:
- Публичные: `/` `/menu` `/delivery` (корзина/оформление) `/booking` `/about` `/contacts`
- Аутентификация: `/login` `/register` `/account`
- Панель персонала (доступ по роли): `/admin` `/kitchen` `/courier`

## 5. Оформление доставки `/delivery` (корзина → заказ)

Форма оформления (для авторизованного пользователя часть полей подтягивается из профиля):
- Состав заказа (позиции, количество, сумма по каждой, **итого**)
- **Адрес доставки** (обязательно)
- **Телефон** (обязательно)
- **Имя**
- **Способ оплаты**: Наличными / Картой курьеру / Kaspi
- **Комментарий** (опционально)
- Кнопка **«Оформить заказ»**

При отправке: создаётся запись в `orders` со статусом `new` и авто-номером заявки (напр. `#00042`), позиции — в `order_items`. Пользователю показываем экран «Заявка №… принята, ожидайте звонка/доставки» + текущий статус.

> Требование заказчика: всё делается **не выходя с сайта** — регистрация/вход, оформление и отслеживание статуса прямо в браузере.

## 6. 🔑 Система заказов и роли (ГЛАВНОЕ)

Четыре роли: **customer (клиент)**, **admin (администратор)**, **kitchen (кухня)**, **courier (курьер)**. Роль хранится в профиле пользователя. Персонал (кухня, курьер) создаётся администратором из панели `/admin` (или админ назначает роль зарегистрированному пользователю).

### 6.1. Жизненный цикл заказа (статусы)

| Статус | Название | Кто меняет | Что дальше |
|---|---|---|---|
| `new` | Новая | клиент (создаёт) | админ подтверждает |
| `cooking` | Готовится | админ (передал на кухню) | кухня готовит |
| `ready` | Готов к выдаче | кухня (готово) | курьер забирает |
| `delivering` | В пути | курьер (взял) | доставка |
| `delivered` | Доставлен | курьер | завершено |
| `cancelled` | Отменён | админ | — |

*(Статусы `new`→`cooking` можно объединить, если админ сразу отправляет на кухню одним действием «Принять и отправить на кухню».)*

### 6.2. Панель администратора `/admin`

- Живая (realtime) лента **новых** заявок: № заявки, время, клиент, телефон, адрес, состав, сумма.
- Кнопки: **«Принять → на кухню»** (`new`→`cooking`), **«Отменить»** (`→cancelled`).
- Общая доска всех активных заказов по статусам (Kanban-колонки: Новые / Готовятся / Готовы / В пути).
- Управление меню (добавить/скрыть блюдо, изменить цену) — CRUD по `menu_items`.
- Список броней (`reservations`) с подтверждением.
- Создание аккаунтов персонала и назначение ролей.

### 6.3. Панель кухни `/kitchen`

- Видит только заказы в статусе `cooking`.
- Карточка заказа: № заявки, состав (список блюд + количество), комментарий клиента.
- Кнопка **«Готово»** (`cooking`→`ready`).
- Обновляется в реальном времени (новый заказ появляется сам).

### 6.4. Панель курьера `/courier`

- Видит заказы в статусе `ready` (готовые к выдаче) — это и есть «чек» от кухни.
- Карточка = **чек курьеру**: № заявки, состав, **итоговая сумма**, **адрес доставки**, **телефон клиента**, способ оплаты.
- Кнопки: **«Взял в доставку»** (`ready`→`delivering`) → **«Доставлен»** (`delivering`→`delivered`).
- Кнопка «Позвонить» (ссылка `tel:`) и «Открыть на карте» (ссылка на карты по адресу).

### 6.5. Realtime

Все три панели подписаны на изменения таблицы `orders` через **Supabase Realtime** — заказ автоматически перескакивает между экранами без перезагрузки. Это ключевая механика.

## 8. База данных (Supabase / Postgres)

```
establishments(
  id, name, address, hours
)

profiles(
  id (=auth.users.id), role ('customer'|'admin'|'kitchen'|'courier'),
  name, phone, created_at
)

menu_items(
  id, name, description, price int, category,
  image_url, establishment_id, is_available bool, created_at
)

orders(
  id, order_number text,          -- напр. #00042
  customer_id, status text,       -- new|cooking|ready|delivering|delivered|cancelled
  total int, address text, phone text, customer_name text,
  payment_method text, comment text,
  courier_id, created_at, updated_at
)

order_items(
  id, order_id, menu_item_id,
  name_snapshot text, price_snapshot int, quantity int
)

reservations(
  id, name, phone, date, time, guests int,
  establishment_id, status ('new'|'confirmed'|'cancelled'), created_at
)
```

**Аутентификация и доступы:**
- Supabase Auth (email+пароль).
- **Row Level Security:**
  - клиент видит и создаёт только свои заказы;
  - admin — все заказы, меню, брони, профили;
  - kitchen — читает заказы `cooking`, меняет на `ready`;
  - courier — читает `ready`/`delivering`, меняет статус доставки.

## 10. (Опционально) Telegram-уведомления — фаза 2

Слой поверх основного веб-приложения, чтобы персонал не пропускал заказы:
- **Supabase Database Webhook** (или Edge Function) на INSERT/UPDATE в `orders`.
- Через **Telegram Bot API** бот пишет в группы:
  - новая заявка / «Готовится» → группа **«Админ/Кухня»**;
  - статус `ready` → группа **«Курьеры»**.
- Текст сообщения = № заявки, состав, сумма, адрес, телефон, способ оплаты.
- Не заменяет веб-панель — только дублирует уведомлением. Всё управление остаётся на сайте.
